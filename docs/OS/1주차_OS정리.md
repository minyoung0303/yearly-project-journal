# 1주차 – OS 기본 & Process / Thread (면접 대비 정리)

> **목표**
> 운영체제의 역할을 명확히 이해하고, Process / Thread 개념을 *면접에서 구조적으로 설명*할 수 있다.

---

## 1. 운영체제(OS)란?

### 1️⃣ 정의

운영체제(OS, Operating System)는 **하드웨어와 사용자 프로그램 사이에서 중재 역할**을 수행하는 시스템 소프트웨어이다.

사용자 프로그램은 직접 하드웨어에 접근하지 않고, 운영체제를 통해 자원을 요청하고 사용한다.

---

### 2️⃣ 운영체제의 핵심 역할

1. **자원 관리(Resource Management)**

   - CPU 스케줄링
   - 메모리 할당 및 회수
   - 디스크, 네트워크 등 I/O 관리

2. **프로세스 관리(Process Management)**

   - 프로세스 생성 / 종료
   - 상태 관리 (Ready, Running, Waiting)
   - Context Switching 수행

3. **보안 및 보호(Protection & Security)**

   - 프로세스 간 메모리 침범 방지
   - 사용자 모드 / 커널 모드 분리

4. **편의성 제공(Abstraction)**

   - 파일 시스템 제공
   - 하드웨어를 추상화하여 개발 난이도 감소

---

### 🎤 면접 한 줄 요약

> 운영체제는 CPU, 메모리 같은 하드웨어 자원을 효율적이고 안전하게 관리하며, 사용자 프로그램과 하드웨어 사이의 중재자 역할을 하는 시스템 소프트웨어입니다.

---

## 2. Process (프로세스)

### 1️⃣ 정의

**프로세스(Process)**란 **실행 중인 프로그램**이다.

- 프로그램: 디스크에 저장된 정적인 파일
- 프로세스: 메모리에 올라가 CPU를 할당받아 실행 중인 상태

---

### 2️⃣ 프로세스의 메모리 구조

각 프로세스는 **독립적인 가상 메모리 공간**을 가진다.

```
Code 영역  : 실행할 명령어
Data 영역  : 전역 변수, 정적 변수
Heap 영역  : 동적 할당 메모리
Stack 영역 : 함수 호출, 지역 변수
```

---

### 3️⃣ PCB (Process Control Block)

운영체제는 각 프로세스를 관리하기 위해 **PCB**를 유지한다.

**PCB에 저장되는 정보**

- PID (Process ID)
- 프로세스 상태 (Ready, Running, Waiting)
- 프로그램 카운터(PC)
- 레지스터 정보
- 스케줄링 정보
- 메모리 관련 정보

➡️ **프로세스의 모든 상태 정보는 PCB에 저장된다.**

---

### 4️⃣ 프로세스의 특징

- 프로세스 간 메모리는 **기본적으로 공유되지 않음**
- 하나의 프로세스가 비정상 종료되어도 다른 프로세스에 영향이 적음
- 통신을 위해서는 **IPC(Inter-Process Communication)** 필요

---

### 🎤 예상 면접 질문

**Q. 프로세스란 무엇인가요?**

> 실행 중인 프로그램이며, 운영체제로부터 자원을 할당받아 독립된 메모리 공간에서 실행되는 작업 단위입니다.

---

## 3. Thread (스레드) ⭐ 매우 중요

### 1️⃣ 정의

**스레드(Thread)**는 **프로세스 내에서 실제로 실행되는 작업의 단위**이다.

- 하나의 프로세스는 하나 이상의 스레드를 가질 수 있다.
- CPU 스케줄링의 기본 단위는 스레드이다.

---

### 2️⃣ 스레드의 메모리 구조

스레드는 프로세스의 자원을 공유한다.

| 구분               | 설명                 |
| ------------------ | -------------------- |
| Stack              | 스레드마다 개별 소유 |
| Code / Data / Heap | 프로세스 내에서 공유 |

➡️ **Stack만 독립적이고 나머지는 공유**

---

### 3️⃣ 스레드의 장점

1. **생성 및 종료 비용이 낮음**
2. **Context Switching 비용이 적음**
3. 자원 공유로 인한 **빠른 통신**
4. 멀티코어 환경에서 **병렬 처리 가능**

---

### 4️⃣ 스레드의 단점

- 하나의 스레드 오류가 프로세스 전체에 영향을 줄 수 있음
- 공유 자원으로 인한 **동기화 문제(Race Condition)** 발생 가능
- Deadlock 위험

---

## 4. Process vs Thread 비교

| 구분              | Process       | Thread                |
| ----------------- | ------------- | --------------------- |
| 실행 단위         | 프로그램 단위 | 프로세스 내 작업 단위 |
| 메모리            | 독립          | 공유                  |
| 생성 비용         | 큼            | 작음                  |
| Context Switching | 비용 큼       | 비용 작음             |
| 통신              | IPC 필요      | 메모리 직접 접근      |
| 안정성            | 높음          | 상대적으로 낮음       |

---

### 🎤 면접 핵심 포인트

**Q. 프로세스와 스레드의 가장 큰 차이는 무엇인가요?**

> 프로세스는 독립된 메모리 공간을 가지는 실행 단위이고, 스레드는 프로세스 내에서 메모리를 공유하며 실행되는 작업 단위입니다.

---

## 5. Context Switching

### 1️⃣ 정의

**Context Switching**이란 CPU가 현재 실행 중인 프로세스(또는 스레드)의 상태를 저장하고, 다른 프로세스의 상태를 복원하여 실행을 전환하는 과정이다.

---

### 2️⃣ 왜 필요한가?

- 멀티태스킹 환경에서 여러 프로세스를 동시에 실행하는 것처럼 보이기 위해
- CPU 자원을 공정하게 분배하기 위해

---

### 3️⃣ Context Switching 과정

1. 현재 실행 중인 프로세스의 레지스터 상태를 PCB에 저장
2. 다음 실행할 프로세스의 PCB에서 상태를 복원
3. CPU 제어권 전환

---

### 4️⃣ 비용 관점

- Context Switching은 **오버헤드**가 발생
- 프로세스 > 스레드 순으로 비용이 큼

---

### 🎤 예상 면접 질문

**Q. Context Switching이란 무엇인가요?**

> CPU가 현재 실행 중인 작업의 상태를 저장하고 다른 작업의 상태를 복원하여 실행을 전환하는 과정입니다.

---

## 6. 실무 & 웹 서버 연결

- 웹 서버는 다수의 클라이언트 요청을 처리해야 함
- 멀티프로세스 / 멀티스레드 모델 사용

**예시**

- 하나의 요청 = 하나의 스레드
- 스레드 풀(Thread Pool)로 성능 최적화

---

## 7. 최종 면접 요약 암기

- 운영체제: 자원 관리 + 중재자
- 프로세스: 실행 중인 프로그램, 독립 메모리
- 스레드: 프로세스 내 실행 단위, 메모리 공유
- PCB: 프로세스 상태 저장 구조체
- Context Switching: 실행 상태 전환 과정
